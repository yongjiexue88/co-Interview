Catch it while you code, not during deployment. Same with performance, tests, and config. Move all that stuff closer to the start, where mistakes are cheap.</p><p class="mb-6 text-gray-300 leading-relaxed">Think of it like catching a bug in a Google Doc before sharing it with your boss. Way less painful.</p><h3 class="text-xl font-bold mb-6 text-white">20. That’s continuous testing. It catches problems while you're still in the zone, not three days later when you’ve forgotten what that function even does.</p><p class="mb-6 text-gray-300 leading-relaxed">It’s like having a personal editor for your code, except they work 24/7 and don’t need coffee breaks.</p><h3 class="text-xl font-bold mb-6 text-white">22. It’s the software version of “set it and forget it.”</p><p class="mb-6 text-gray-300 leading-relaxed">Used right, it saves hours. Used wrong, and you’ll automate bugs at lightning speed. So write good tests.</p><h3 class="text-xl font-bold mb-6 text-white">23. Centralized vs. Distributed: Who's Holding the Truth?</h3><p class="mb-6 text-gray-300 leading-relaxed">When I was ramping up for my Amazon interview, I thought I had Git down cold… until they hit me with a version control curveball. “What’s the tradeoff between centralized and <a href="https://www.researchgate.net/publication/306011858_THE_ROLE_OF_DISTRIBUTED_VERSION_CONTROL_SYSTEMS_IN_TEAM_COMMUNICATION_AND_LEARNING_SYSTEM" target="_self" class="text-white underline decoration-gray-400 underline-offset-2 hover:decoration-gray-600">distributed VCS</a>?” Uh, Git good, SVN bad?</p><p class="mb-6 text-gray-300 leading-relaxed">Here’s the breakdown minus the resume buzzwords:</p><h4 class="text-lg font-bold mb-6 text-white">Centralized Version Control (think Subversion)</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">One server holds everything. No local backups.</li><li class="text-gray-300">You go down with the ship if that server crashes.</li><li class="text-gray-300">Slightly simpler for small teams, but brittle.</li></ul><h4 class="text-lg font-bold mb-6 text-white">Distributed Version Control (like Git)</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">Everyone has a full copy of the repo on their machine.</li><li class="text-gray-300">You can work offline. CI/CD runs faster—no single point of failure.</li><li class="text-gray-300">More branching freedom. More complexity, too.</li></ul><h4 class="text-lg font-bold mb-6 text-white">Interview Tip</h4><p class="mb-6 text-gray-300 leading-relaxed">Be ready to talk tradeoffs. Centralized = easier permissions. Distributed = better for experimentation, branching, and merges. Keep it real.</p><h3 class="text-xl font-bold mb-6 text-white">2. Clone That Repo: What's the Command?</h3><p class="mb-6 text-gray-300 leading-relaxed">If you can't clone a repo without Googling it, you're not ready for Git questions. Just saying.</p><p class="mb-6 text-gray-300 leading-relaxed">git clone https://github.com/user/repo.git</p><p class="mb-6 text-gray-300 leading-relaxed">Or, if you're cool like me and use SSH:</p><p class="mb-6 text-gray-300 leading-relaxed">git clone git@github.com:user/repo.git</p><p class="mb-6 text-gray-300 leading-relaxed">Stuff worth knowing (because they <em class="italic text-gray-300">will</em> ask):</p><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">For CI: use --depth 1 to keep builds light.</li><li class="text-gray-300">Use HTTPS tokens or SSH keys, not your password.</li><li class="text-gray-300">CI agents need creds too don't hardcode them, obviously.</li></ul><h3 class="text-xl font-bold mb-6 text-white">3. Pushing to GitHub: What's the Process?</h3><p class="mb-6 text-gray-300 leading-relaxed">I’ve seen folks freeze on this mid-interview. Here’s the simple version:</p><p class="mb-6 text-gray-300 leading-relaxed">git remote add origin https://github.com/your/repo.git</p><p class="mb-6 text-gray-300 leading-relaxed">git push origin main</p><p class="mb-6 text-gray-300 leading-relaxed">If your default branch is still called master, fix that.</p><p class="mb-6 text-gray-300 leading-relaxed">More edge cases:</p><p class="mb-6 text-gray-300 leading-relaxed">New branch?git push -u origin feature/xyz</p><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">If you get blocked, check for protected branches or required reviews.</li><li class="text-gray-300">Oh, and CI might yell at you for skipping tests. Don’t skip the tests.</li></ul><h3 class="text-xl font-bold mb-6 text-white">4. Bare Repo vs. Regular Init: Why It Matters</h3><p class="mb-6 text-gray-300 leading-relaxed">This one tripped me up during a mock with a Google engineer.</p><h4 class="text-lg font-bold mb-6 text-white">Regular git init</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">Creates a working directory you can edit.</li><li class="text-gray-300">Stores Git data in a .git/ folder.</li></ul><h4 class="text-lg font-bold mb-6 text-white">Bare git init --bare</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">No working files. Just the Git history.</li><li class="text-gray-300">Used for remote repos (CI/CD, GitLab server, etc.)</li></ul><p class="mb-6 text-gray-300 leading-relaxed">Basically, if it’s a shared repo on a server that others push to, it should be bare. If it’s your personal playground, regular init is fine.</p><h3 class="text-xl font-bold mb-6 text-white">5. Oops: You Pushed a Bad Commit. Now What?</h3><p class="mb-6 text-gray-300 leading-relaxed">This is where people start sweating. Here's the non-destructive fix:</p><p class="mb-6 text-gray-300 leading-relaxed">git revert &lt;commit-hash&gt;</p><p class="mb-6 text-gray-300 leading-relaxed">That creates a new commit that undoes the mistake.</p><p class="mb-6 text-gray-300 leading-relaxed">What not to do:</p><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">git reset --hard on shared branches. That’s how you make enemies.</li><li class="text-gray-300">Force-pushing to main unless you're into chaos engineering.</li></ul><h4 class="text-lg font-bold mb-6 text-white">Interview Tip</h4><p class="mb-6 text-gray-300 leading-relaxed">Mention feature flags and rollout toggles. Git isn't your only rollback tool.</p><h3 class="text-xl font-bold mb-6 text-white">6. Fetch vs. Pull: What’s the Actual Difference?</h3><p class="mb-6 text-gray-300 leading-relaxed">This isn’t a trick question, but people overthink it.</p><h4 class="text-lg font-bold mb-6 text-white">git fetch</h4><p class="mb-6 text-gray-300 leading-relaxed">Grabs the latest commits, but doesn't touch your working files like reading the news, but not reacting.</p><h4 class="text-lg font-bold mb-6 text-white">git pull</h4><p class="mb-6 text-gray-300 leading-relaxed">Fetch + merge (or rebase, depending on your config). It updates your local branch.</p><p class="mb-6 text-gray-300 leading-relaxed">For interviewers:</p><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">Say fetch is safer if you're not ready to merge.</li><li class="text-gray-300">Rebase is cleaner for history. Git Stash: What Do You Do When You’re in the Middle of Something and Suddenly Get Pulled Into Something Else?</h3><p class="mb-6 text-gray-300 leading-relaxed">Happens to the best of us: you're halfway through writing code that’s <em class="italic text-gray-300">almost</em> working... then Slack lights up. “Can you switch to this hotfix branch real quick?”</p><p class="mb-6 text-gray-300 leading-relaxed">Now, if you commit your half-broken code just to stash it somewhere, that’s going to haunt you later in code review. Ask me how I know.</p><p class="mb-6 text-gray-300 leading-relaxed">That’s where git stash comes in clutch. It lets you temporarily shelf your changes <em class="italic text-gray-300">without</em> committing like a messy draft shoved in a drawer, so you can come back to it when the fire drill is over.</p><h4 class="text-lg font-bold mb-6 text-white">What Interviewers Expect Here</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">Be fluent with git stash, git stash list, git stash apply, git stash pop, git stash drop.</li><li class="text-gray-300">Mention git stash branch &lt;name&gt; — underrated move to spin off into a fresh branch.</li><li class="text-gray-300">Bonus points: Talk about how git stash -u includes untracked files (which can mess with CI if you’re not careful)</li></ul><h3 class="text-xl font-bold mb-6 text-white">8. Git Branching: Yeah, You’re Supposed to Use Them. Like, Properly.</h3><p class="mb-6 text-gray-300 leading-relaxed">Look, when I first started, I pushed everything straight to main like a gremlin. Nobody told me that branching was how you <em class="italic text-gray-300">don’t</em> wreck production.</p><p class="mb-6 text-gray-300 leading-relaxed">So, you want to add a feature? Branch it. Bugfix? Branch it. Tiny UI tweak? Honestly... still, branch it.</p><p class="mb-6 text-gray-300 leading-relaxed"><a href="https://www.researchgate.net/publication/388531640_Git_Branching_and_Release_Strategies" target="_self" class="text-white underline decoration-gray-400 underline-offset-2 hover:decoration-gray-600">Git branching</a> is just controlled chaos; it gives you the space to build without breaking other people’s work.</p><h4 class="text-lg font-bold mb-6 text-white">If They Press You On This In Interviews</h4><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">git branch feature/x to create</li><li class="text-gray-300">git checkout -b feature/x to create <em class="italic text-gray-300">and</em> switch (shorthand flex)</li><li class="text-gray-300">git merge or a pull request to merge back</li><li class="text-gray-300">Know your flavors: feature branching vs trunk-based development</li><li class="text-gray-300">Be ready to talk about protected branches, naming patterns, and CI checks on PRs</li></ul><h3 class="text-xl font-bold mb-6 text-white">9. Git Merge vs Rebase: This One <em class="italic text-gray-300">Will</em> Come Up</h3><p class="mb-6 text-gray-300 leading-relaxed">Okay, real talk, I messed this up in my first technical interview at Meta.</p><p class="mb-6 text-gray-300 leading-relaxed">Here’s the scenario they gave me:</p><p class="mb-6 text-gray-300 leading-relaxed">You're working on a feature branch. In the meantime, other people push changes to main. What now?</p><p class="mb-6 text-gray-300 leading-relaxed">You’ve got two options:</p><h4 class="text-lg font-bold mb-6 text-white">Merge</h4><p class="mb-6 text-gray-300 leading-relaxed">Pull those changes into your branch with git merge main. Safe, easy, creates a merge commit.</p><h4 class="text-lg font-bold mb-6 text-white">Rebase</h4><p class="mb-6 text-gray-300 leading-relaxed">Rewrite your branch so it <em class="italic text-gray-300">looks like</em> it was built on top of the latest leading clean history, but riskier.</p><p class="mb-6 text-gray-300 leading-relaxed">How to not sound like you memorized a cheat sheet:</p><ul class="mb-6 space-y-2 pl-6"><li class="text-gray-300">Say that merge keeps history intact, which teams may prefer for traceability</li><li class="text-gray-300">Rebase gives a linear history, great for local cleanup before pushing.</li><li class="text-gray-300">Never rebase shared branches. Ever.</li><li class="text-gray-300">Mention git rebase -i for squashing commits. CI bots love tidy commits.</li></ul><h3 class="text-xl font-bold mb-6 text-white">10. Use git show --name-only &lt;commit&gt; or git diff --name-only &lt;commit&gt;^ &lt;commit&gt;</p><p class="mb-6 text-gray-300 leading-relaxed">Want to know if files were added, deleted, or modified? --name-status is your friend.</p><p class="mb-6 text-gray-300 leading-relaxed">This is one of those areas where interviewers check if you're a click-everywhere type or someone who knows how to script their workflow.</p><h3 class="text-xl font-bold mb-6 text-white">11. Merge Conflicts: The Messy Middle of Teamwork</h3><p class="mb-6 text-gray-300 leading-relaxed">Let me paint a picture. You and your teammate both edit the same line in a file. You push first. They try to merge later, but there is a merge conflict.</p><p class="mb-6 text-gray-300 leading-relaxed">No magic fix. You’ve got to pick what stays and what gets binned manually.</p><p class="mb-6 text-gray-300 leading-relaxed">I usually open up the file, look for those angry &lt;&lt;&lt;&lt;&lt;&lt;&lt;, =======, and &gt;&gt;&gt;&gt;&gt;&gt;&gt; markers, and start cleaning.</p><h4 class="text-lg font-bold mb-6 text-white">GitHub GUI? Git Bisect: How to Catch the Commit That Broke Everything</h3><p class="mb-6 text-gray-300 leading-relaxed">I call this “bug hunting with a lie detector.”</p><p class="mb-6 text-gray-300 leading-relaxed">git bisect is what you run when <em class="italic text-gray-300">something</em> broke, and you have <strong class="font-bold text-white">no idea</strong> which commit caused it.</p><p class="mb-6 text-gray-300 leading-relaxed">You tell Git:</p><p class="mb-6 text-gray-300 leading-relaxed">git bisect start</p><p class="mb-6 text-gray-300 leading-relaxed">git bisect bad</p><p class="mb-6 text-gray-300 leading-relaxed">git bisect good &lt;known-good-commit&gt;</p><p class="mb-6 text-gray-300 leading-relaxed">Then it keeps narrowing it down, binary search style. Three Variables That Control Recursion And Inheritance In Nagios</h3><p class="mb-6 text-gray-300 leading-relaxed">Nagios doesn’t really “think” in objects the way a Java dev would, but its config system borrows the idea of reusable building blocks.
